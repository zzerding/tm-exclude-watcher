# ADR-0001: 多进程直接访问数据库

**状态：** 已接受  
**日期：** 2026-06-10  
**决策者：** Doctor Biz

## 上下文

tm-watcher 有多个并发访问源：
- 守护进程（FSEvents 处理线程、定期清理定时器）
- CLI 命令（用户执行 `scan`、`clean`、`list`）

需要决定并发控制策略，以防止数据库竞态和数据损坏。

## 决策

**采用多进程 + 数据库事务模型：**
- CLI 命令直接操作 SQLite 数据库，不通过 IPC 与守护进程通信
- 依赖 SQLite 的文件锁机制保证并发安全
- 使用 `INSERT OR IGNORE` 和 `UNIQUE` 约束防止重复记录
- 守护进程内部用单线程 async runtime 避免线程竞争

## 备选方案

### 方案 A：IPC + 单一写入者（未采用）
- CLI 命令通过 Unix socket 与守护进程通信
- 只有守护进程能写数据库
- CLI 发送请求，守护进程执行并返回结果

**拒绝理由：**
- 增加复杂度：需要实现 IPC 协议、请求队列、超时处理
- 守护进程未运行时 CLI 无法工作（需要额外的"降级模式"）
- 调试困难：进程间通信的 bug 更难追踪

## 后果

### 正面
- **实现简单：** SQLite 自带并发控制，零额外代码
- **独立性：** CLI 命令可独立于守护进程运行（例如 `scan` 可在守护进程停止时使用）
- **调试友好：** 每个进程独立操作，日志清晰，易于追踪

### 负面
- **文件锁争用：** 高频并发写入时可能遇到 `SQLITE_BUSY`（但 tm-watcher 的写入频率很低，实际不是问题）
- **无法实现复杂事务：** 如果未来需要跨进程的原子操作，当前架构无法支持

## 风险缓解

- 设置合理的 SQLite `busy_timeout`（例如 5 秒），减少 `SQLITE_BUSY` 错误
- 如果未来需要跨进程事务，可以在那时重构为 IPC 模型（数据量小，迁移成本可控）
