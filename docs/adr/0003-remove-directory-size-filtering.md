# ADR-0003: 移除目录大小过滤功能

**状态：** 已接受  
**日期：** 2026-06-10  
**决策者：** Doctor Biz

## 上下文

PRD 最初提议 `min_directory_size_mb` 配置项，只排除大于阈值的目录。

**原始意图：**
- 避免排除"小的无关紧要的目录"
- 让用户自主控制排除策略

**示例配置：**
```toml
[behavior]
min_directory_size_mb = 100  # 只排除 >= 100 MB 的目录
```

## 决策

**移除 `min_directory_size_mb` 过滤功能。**

只要目录名匹配规则（`node_modules`、`target` 等），无论大小都排除。

**但保留 `size_bytes` 字段用于统计和展示。**

## 理由

### 1. 语义错误
- 10 MB 的 `node_modules` 和 500 MB 的 `node_modules` 本质相同：都是可重现的依赖，不需要版本历史
- "大小"不是判断"是否应该备份"的合理标准
- 正确的标准是"内容性质"（源代码 vs 构建产物）

### 2. 性能成本
- 递归计算目录大小是昂贵操作（`node_modules` 可能有数万个文件）
- 每次检测到目录都要计算大小，会成为性能瓶颈
- 在确认延迟后计算依然需要 I/O，拖慢响应速度

### 3. 增加复杂度
- 需要额外配置项、文档说明、边界值处理（恰好 100 MB？）
- 用户需要理解并调整阈值（100 MB 合理吗？50 MB？500 MB？）
- 增加测试覆盖范围（大目录、小目录、边界值）

## 备选方案

### 方案 A：保留大小过滤（未采用）
**拒绝理由：** 见上述三点

### 方案 B：完全不记录大小（未采用）
**拒绝理由：**
- 统计功能有价值："tm-watcher 为你节省了 5 GB 备份空间"
- 帮助用户发现异常大的依赖目录（例如 2 GB 的 `node_modules` 可能有问题）
- 用户可能想手动清理最大的几个目录

### 方案 C：记录大小但不过滤（已采用）
- 排除时计算并记录初始大小
- 定期清理时更新大小（维持新鲜度）
- `tm-watcher list` 可按大小排序，辅助用户决策

## 后果

### 正面
- **简化配置：** 用户无需理解和调整大小阈值
- **性能更好：** 排除决策立即做出，无需等待大小计算（但依然在后台计算用于统计）
- **语义正确：** 排除策略基于"内容类型"而非"大小"

### 负面
- **灵活性降低：** 用户无法通过大小阈值控制排除行为（但这种需求场景本身就不合理）
- **可能排除小目录：** 例如只有 1 MB 的 `node_modules`（但排除它也无害，占用的备份空间和 I/O 都微不足道）

## 补偿机制

未来如果有用户真的需要"不排除某些特定目录"，可通过白名单机制实现（v0.3+）：
```toml
whitelist_paths = [
    "~/Code/special-project/node_modules"  # 这个项目的依赖需要备份
]
```

这比大小阈值更精确、更可控。
